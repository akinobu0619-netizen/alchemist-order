# SPEC: 週替わりシード塔（F2）

策定: 2026-07 / 担当: Fable(設計)→Opus(実装=O9)→Sonnet(検証=S3系)
前提: [SPEC_RNG_REPLAY.md](SPEC_RNG_REPLAY.md)（F1/O3・シードRNG基盤）。運用は [PROMO_PLAN.md](PROMO_PLAN.md) の「月曜=塔告知」リズムと接続。

## 0. 一文サマリ

**全員が同じ週は同じ敵・同じ乱数と戦う、毎週リセットされる競争モードの塔。**
運の引き直しは週シード固定で原理的に無効化され、順位を分けるのは「編成と操作の改善」だけになる。
既存の「試練の塔（エンドレス・自己ベスト）」は残し、その隣に競争用の「今週の塔」を追加する。

## 1. なぜ作るか（狙い）

- **再訪の理由を作る**: 「毎週月曜、塔が変わる」= フォロー・ブックマークの動機。PROMO_PLANの週次リズムの主役。
- **公平な競争**: 週シード固定なので、他人と同条件。スクショ自慢が「運自慢」でなく「攻略自慢」になる。
- **将来のサーバ順位表(F5/O10)の入口**: v1はローカル+ハッシュタグ運用、v2でリプレイlog提出→サーバ再シミュ。
- **既存資産の再利用**: 塔の戦闘ループ・シード基盤・シェアカードは実装済み。追加は「シード源・敵編成・スコア・週モディファイア・UI」の5点に閉じる。

## 2. スコープ（v1 = O9で実装する範囲）

含む: 週シード導出／決定論の敵編成テーブル／スコア式と自己週間ベスト保存／週替わりモディファイア／エントリUIとシェア連携。
含まない（v2以降）: サーバ順位表・リプレイlog提出（F5/O10）／パーティ正規化（§4のv2案）／観戦再生。

## 3. 週シードと週境界

```
weekKey  = ISO年-ISO週  例 "2026-W28"          // §3.1のJST基準で算出
runSeed  = `AO1|wtower|${weekKey}`              // 全プレイヤー共通(SPEC_RNG_REPLAY §2の予約形式)
floorSeed= makeRng(runSeed).fork(`floor:${n}`)  // 既存塔と同じfork規約
battleRng= floorSeed.fork('battle')
```

### 3.1 週境界（重要・タイムゾーン）

- **週の切り替えは月曜 00:00 JST（Asia/Tokyo）**。主対象が国内のため。ISO週(月曜始まり)と一致。
- 実装は「UTC時刻 + 9時間」した日付からISO週番号を算出する純関数 `weekKeyOf(nowMs: number): string` を `src/engine/` に置く（`Date.now()`はこの1関数に閉じ込め、他はここ経由）。
- 表示は「今週の塔（〜7/12 23:59まで）」のように締切をJSTで明示。

### 3.2 週内バランス凍結

- SPEC_RNG_REPLAY §7に従い、**週の途中で敵編成・技・種族値・相性表を変更しない**（スコアの土台が動くため）。
- やむを得ず修正した場合はその週を「参考記録」に降格（順位表運用時）。データ変更は日曜深夜以降にまとめる運用とする。

## 4. パーティとフェアネス

- **v1: パーティは正規化しない**（プレイヤーの実個体・実Lv・技・持ち物・talentのまま挑む）。
  - 過強編成の抑制は§5の**敵レベル逓増**が担う：敵は階が上がるほど強くなり最終的にLv100帯へ到達、どんな編成もいずれ壁に当たる。差がつくのは「どこまで」+ §6のタイブレーク。
  - これで「育てたコレクションが活きる」ゲームの核を保ちつつ、天井到達者どうしは操作精度で決着する。
- **満タン開始**（既存 `startTower` の `healParty` を踏襲）。道具は所持数から持込、**持込数をログに記録**（SPEC_RNG_REPLAY §5, §8）。
- **フェアネス原則（S3の教訓を内蔵）**: 「シルヴァ初版0.4%」「マレア0%」の轍を踏まない。
  - どの御三家(火/水/風)でも詰まない編成であること。§5の各バンドは**必ず複数タイプを混在**させ、単一タイプの壁を作らない。
  - モディファイア(§7)も特定御三家を一方的に不利にするものは入れない。
  - ボス級floorのステータスは**Sonnetのsim_balance.mjsで勝率を測ってから確定**（暫定値のまま出さない）。

### v2メモ（正規化案・実装しない）
競争の純度を上げるなら「入場時に全パーティを固定Lv(例30)へ正規化（技/talent/持ち物は保持）」で完全パズル化できる。ただし育成RPGの手触りを削ぐため、順位表が本格稼働し「ガチ勢向け別レギュ」需要が出てから検討。

## 5. 敵編成テーブル（決定論・逓増・タイプ混在）

現状 `towerConfig(floor, seed)` は `min=max=4+floor*2`・pool未指定（全非伝説stage≤2からランダム）。週塔ではこれを**キュレーションされたバンド**に差し替える。

| 階帯 | Lv帯 | プール方針 | 特殊 |
|---|---|---|---|
| 1–4 | 8–16 | stage1中心・3タイプ以上混在 | 足慣らし |
| 5 | +2補正 | ミニボス(talent3・stage2) | 5階ごと=節目 |
| 6–9 | 18–26 | stage1-2混在・毒/雷など状態技持ち登場 | 状態異常の駆け引き |
| 10 | 固定種 | **守護者floor**(週替わりで種を変える・talent5) | §7モディファイアの象徴 |
| 11–19 | 28–44 | stage2中心・相性読みが必須 | — |
| 20 | 固定種 | 準ボス(伝説級手前・talent6・持ち物付き) | 到達者を絞る関門 |
| 21+ | 46→100 | stage2-3・レア枠15%・全タイプ | 天井(Lv100帯で自然な壁) |

- 各階の種/Lv/talent/持ち物は `floorSeed` から**決定論的に**引く（同週=全員同一）。
- 「3タイプ以上混在」はプール構築時にタイプ集合サイズを検査する制約として実装（単色バンド禁止）。
- 守護者floor(10)と準ボス(20)は「その週の固定種」を週シードから選ぶ→毎週見せ場が変わる。

## 6. スコア式（順位が付けられること）

プレイヤーには「何階？」で通じる素朴さを保ちつつ、天井到達者を割るタイブレークを持たせる。

```
主指標   : reached（制覇階数）              ……プレイヤー表示はこれ
タイブレーク（同 reached 内の優劣、辞書式）:
  1. 消費ターン総数が少ない          （速い＝上位）
  2. 使用道具数が少ない              （省エネ＝上位）
  3. 最終パーティ残HP合計%が高い     （余力＝上位）
内部スコア（順位表用の単一数値）:
  score = reached*1_000_000 - turns*100 - itemsUsed*1_000 + hpPct
        （hpPct=残HP%合計 0..600。係数は「上位指標が必ず優先」になるよう桁で分離）
```

- 表示は常に「◯階制覇」。`score`は順位表・シェアカードの内部比較にのみ使用（数字の説明責任を負わない）。
- ログ(SPEC_RNG_REPLAY §5)の `result` を `{ reached, turns, itemsUsed, hpPct, score }` に拡張。

## 7. 週替わりモディファイア（今週の目玉）

毎週1つ、週シードから決定論的に選ぶ小さな一捻り。**理解が一瞬で済む・特定御三家を殺さない**もののみ。

| id | 名称 | 効果 | フェアネス確認 |
|---|---|---|---|
| `swift` | 疾風の週 | 全戦闘で素早さ+段階1相当スタート | 全員一律=公平 |
| `frugal` | 節制の週 | 道具持込が半分（切り上げ） | 全員一律 |
| `bounty` | 豊穣の週 | レア枠出現率2倍(見た目のご褒美・強さ不変) | 有利化のみ |
| `elite` | 精鋭の週 | 各floorのtalentが+2 | 全タイプ均等に強化 |
| `sudden` | 短期決戦の週 | 各floorの敵最大HP-15%だが自軍も-15% | 対称 |
| `guardian` | 守護者強化の週 | 10/20階のボスのみtalent+2 | 局所・タイプ非依存 |

- 導出: `modifier = MODS[ makeRng(runSeed).fork('mod').int(0, MODS.length-1) ]`。
- **禁止例**（入れない）: 「今週は水タイプ強化」等の単一タイプ優遇＝御三家間の不公平を生むため不可。
- UIとシェアカードに「今週: 疾風の週」を明記。告知文(PROMO ③)にも差し込める。

## 8. 保存とリトライ

- **リトライ無制限**（週シード固定＝運の引き直しは無効。試行の価値は操作改善のみ＝SPEC_RNG_REPLAY §2）。
- ゲーム状態に週間ベストを追加（既存 `towerBest`(エンドレス)とは別枠）:
```ts
// types.ts GameState へ
weeklyTower?: { week: string; reached: number; score: number }  // 現在週のベストのみ保持
weeklyHistory?: { week: string; reached: number }[]             // 過去週の自己記録(任意・最大12件)
```
- 週が変わったら `weeklyTower` はリセット（初挑戦で新規作成）。切替時に旧週を `weeklyHistory` へ退避。
- 自己ベスト更新時のみ `tower_end` 相当の計測イベント（`weekly_tower_end` を追加・§10）。

## 9. UI / 導線

- ワールド/ホームの既存「試練の塔」ボタンの隣に **「今週の塔」** を追加。
  - 表示: 「今週の塔　残り3日 / 今週のベスト 12階 / 疾風の週」。
  - エンドレス塔ボタンのラベルを「試練の塔（自己ベスト）」に、週塔を「今週の塔（競争）」に整理。
- クリア/敗退ダイアログとシェアカード([ShareCard.tsx](../src/screens/ShareCard.tsx))に週ラベル＋モディファイア名を追加。
  - `ShareData` に `weekKey?: string; modifierName?: string` を追加、カード見出しを「今週の塔(2026-W28) 12階制覇！」に。
- ハッシュタグ運用(v1)の導線: シェア画像に週ラベルが入る＝Xで同週プレイヤーの比較が成立する。

## 10. 計測（O1/analytics連携）

- `weekly_tower_start` `{ week, modifier }`
- `weekly_tower_end` `{ week, modifier, reached, score, best_updated }`
- 週次KPI（PROMO §KPI）: 週塔の参加者数・中央到達階・シェア数。月曜告知の反応をここで測る。

## 11. 実装アーキテクチャ（O9への作業指針）

```
src/engine/week.ts        … 新規。weekKeyOf(nowMs) / weekSeedOf(weekKey) / pickModifier(runSeed)
                            (Date.now依存はここに隔離)
src/game/towerBands.ts    … 新規。§5のバンド定義＋floorSeedから決定論編成する純関数
                            makeWeeklyFloor(runSeed, floor, modifier) => BattleConfig
src/App.tsx               … startWeeklyTower() 追加（startTower を一般化）。tower stateに mode/week/modifier を持たせる
                            towerConfig を「エンドレス用」と「週用(towerBands呼出)」で分岐
src/screens/Battle.tsx    … モディファイア適用(swift/elite/sudden等)を敵/自軍生成時に反映。既存シードforkはそのまま流用
src/screens/ShareCard.tsx … weekKey/modifierName 表示追加
src/types.ts              … GameState に weeklyTower/weeklyHistory、result拡張、BattleConfigにmodifier
```

- O3β(turnResolver抽出)は**必須前提ではない**（週塔はスコア競争であって、v1の順位はローカル+ハッシュタグ運用のため）。
  ただしサーバ順位表(v2/O10)の再シミュ検証にはO3β/γが要る。**v1はO3β無しで出荷可能**、v2で回収する。

## 12. 受け入れテスト（Sonnet/S3系）

1. **週決定論**: 同一weekKeyでの敵編成生成を2回→全floor完全一致。異なるweekで編成が変わる。
2. **週境界**: 月曜00:00 JST直前/直後でweekKeyが正しく切替（UTC+9境界のテストケース。年跨ぎISO週も確認）。
3. **フェアネス**: 火/水/風の各代表編成でsim_balance.mjsを回し、**どの御三家でも一定階(例10階)到達率が実用域**（単色バンドが無いこと・特定御三家0%が出ないこと）。S3の轍チェック。
4. **モディファイア対称性**: `sudden`(±15%)等が自他対称に効く。`swift`が全戦闘に適用される。
5. **スコア単調性**: reachedが1多い記録は、他要素に関わらず必ずscoreが上（桁分離の検証）。
6. **保存**: 週跨ぎで `weeklyTower` がリセットされ旧週が `weeklyHistory` に残る。回帰QA(S2)にセーブ入出力を追加。
